## Iteration 4 - US-004: Get check-in roster tool
Started: 2026-03-09T14:19:00Z
Completed: 2026-03-09

### What was built
- get_checkin_roster tool: accepts date, schedule (required), optional location and group filters
- Resolves PersonAliasId → PersonId → Person name via two-step API lookup
- Caches person lookups in a dict within each request to minimize redundant API calls
- Caches location name lookups via LOCATION_NAMES dict (persists across calls)
- Returns roster entries with name, group name, location/room name, category
- Results sorted by category (location bucket) then by person name
- Reuses existing _fetch_occurrences, _fetch_attendances, _fetch_group, _categorize helpers

### Learnings
- Rock RMS PersonAlias/{id} returns PersonId which can then be used with People/{id} for name
- PersonAliasId is the foreign key in Attendance records, not PersonId directly
- Two-step resolution (alias→person) means caching is critical to avoid O(2n) API calls per roster
- LOCATION_NAMES dict serves dual purpose: hardcoded known names + dynamic cache for unknown location IDs
Completed: 2026-03-09T14:20:47Z

## Iteration 5 - US-005: Get group roster tool
Started: 2026-03-09T14:20:49Z
Completed: 2026-03-09

### What was built
- get_group_roster tool: accepts group_name (string) or group_id (int)
- Searches groups via GET /api/Groups?$filter=substringof('{name}', Name) (OData v3 syntax)
- Fetches group members via GET /api/GroupMembers?$filter=GroupId eq {id}&$expand=Person,GroupRole
- Returns member name, role (Leader/Member), status (Active/Inactive/Pending)
- Includes group metadata: name, description, group type, schedule, location
- Results sorted: Active before Inactive, Leaders before Members, then alphabetical
- Returns single group object if one match, array if multiple matches

### Learnings
- OData v3 uses substringof('{value}', PropertyName) not contains(PropertyName, '{value}') for string contains
- GroupMemberStatus is an enum: 0=Inactive, 1=Active, 2=Pending
- $expand=Person,GroupRole on GroupMembers gives nested person details and role name in one call
- GroupLocations is a collection on Group; first entry's LocationId gives primary location
Completed: 2026-03-09T14:22:37Z

## Iteration 6 - US-006: Attendance trends tool
Started: 2026-03-09T14:22:39Z
Completed: 2026-03-09

### What was built
- get_attendance_trends tool: accepts start_date, end_date, optional schedule, optional location_group
- Iterates over all Sundays in date range, fetches Saturday (preceding day) and Sunday services
- Reuses existing _process_service helper for each date/schedule combo — no duplicated logic
- Returns per-week breakdown with services, categories, and week totals
- Summary stats per service and overall: average, min, max, trend direction
- Trend direction uses linear regression slope as percentage of mean (>2% = increasing, <-2% = decreasing)
- Missing weeks naturally show 0 totals when _process_service returns empty counts
- location_group filter applies per-service to narrow to a single category

### Learnings
- Composing on _process_service avoids duplicating the occurrence/attendance fetch+categorize logic
- Trend calculation: filtering out zero-attendance weeks before computing slope avoids holiday weeks skewing the trend
- _sundays_in_range helper computes first Sunday >= start_date, then steps by 7 days
- Saturday dates are derived as sunday - 1 day, keeping the week anchored on Sunday
